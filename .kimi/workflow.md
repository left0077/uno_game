# Kimi 工作流指南

## 开始任务前的检查清单

### 1. 查找现有约定
```bash
# 查找枚举/常量定义
grep -r "enum\|const.*=" --include="*.ts" src/ | grep -v node_modules

# 查找配置文件
grep -r "config\|settings" --include="*.ts" --include="*.json" | head -20

# 查找类似的实现
grep -r "emit\|on(" --include="*.ts" | head -20
```

### 2. 检查 AGENTS.md
- 项目根目录是否有 `AGENTS.md`
- 相关子目录是否有 `AGENTS.md`
- 文件是否提到相关规范

### 3. 检查 README
- 是否有架构说明
- 是否有命名规范
- 是否有开发指南

## 修复问题的正确流程

### 场景：Socket 事件命名不一致

❌ **错误做法**
1. 创建一个全新的标准文档
2. 编写复杂的设计方案
3. 最后才改代码

✅ **正确做法**
1. **查现有代码**
   ```bash
   grep -n "enum.*Events" server/src/shared/index.ts
   # 发现 SocketEvents 枚举
   ```

2. **对齐现有标准**
   - 直接使用 `SocketEvents.XXX`
   - 删除硬编码字符串
   - 不改枚举定义（除非必要）

3. **测试验证**
   - 构建测试
   - 运行现有测试
   - 不创建新测试类型（除非必要）

## 禁止事项

- [ ] **禁止** 创建新的规范文档（除非项目完全没有）
- [ ] **禁止** 修改现有测试文件的结构
- [ ] **禁止** 重命名现有的函数/变量（除非必要）
- [ ] **禁止** 添加新依赖（除非必要）
- [ ] **禁止** 修改配置文件（除非必要）

## 允许事项

- [ ] 修复 bug
- [ ] 对齐现有约定
- [ ] 删除废弃代码
- [ ] 添加注释说明
- [ ] 创建最小的修复 PR

## 决策流程

```
发现问题
  ↓
查现有代码是否有约定？
  ├─ 有 → 对齐现有约定
  └─ 无 → 参考类似项目
  ↓
能否改代码解决？
  ├─ 能 → 直接改代码
  └─ 不能 → 才考虑写文档
  ↓
测试验证
  ↓
完成
```

- [ ] 没有修改测试结构
